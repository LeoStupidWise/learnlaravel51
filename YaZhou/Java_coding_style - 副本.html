PHP后台代码风格参考规范

Author：Zoe

Last update：2017/03/21

1 概览

  • PHP代码文件必须以 < ?php 或 < ?= 标签开始。

  • PHP代码文件必须以 不带BOM的 UTF-8 编码。

  • PHP代码中应该只定义类、函数、常量等声明，或其他会产生 从属效应 的操作（如：生成文件输出以及修改.ini配置文件等），二者只能选其一。

  • 命名空间以及类必须符合 PSR 的自动加载规范。

  • 类的命名必须遵循 StudlyCaps 大写开头的驼峰命名规范。

  • 类中的常量所有字母都必须大写，单词间用下划线分隔。

  • 方法名称必须符合 camelCase 式的小写开头驼峰命名规范。

2 命名空间和类

package语句不换行，即package语句写在一行里。

2.2 import语句

2.2.1 import不要使用通配符

即，不要出现类似这样的import语句：import java.util.*;

2.2.2 不要换行

import语句不换行，即每个import语句独立成行。

2.3 类声明

2.3.1 只有一个顶级类声明

每个顶级类都在一个与它同名的源文件中(当然，还包含.java后缀)。即内部类不能和类文件名相同。

2.3.2 类成员顺序

每个类的类成员尽量按 常量 -> 类成员变量 -> 静态代码块 -> 构造方法 -> public方法 -> protected方法 -> private方法 -> 静态方法 的顺序排列。此处根据具体情况可做调整，并不严格要求，但必须遵守先写类成员变量后写成员方法的格式。

2.3.3 重载：永不分离

当一个类有多个构造函数，或是多个同名方法，这些函数/方法应该按顺序出现在一起，中间不要放进其它函数/方法。

3 命名约定

3.1 包名

包名全部小写，连续的单词只是简单地连接起来，不使用下划线。

3.2 类名

  • 类名都以大驼峰式命名法风格编写。

  • 类名通常是名词或名词短语，接口名称有时可能是形容词或形容词短语。建议使用符合该类特定含义的英文名称，如UserDao（表示操作数据库用户表的类）、ProfileAcivity(表示个人信息Activity)。

  • 测试类的命名以它要测试的类的名称开始，以Test结束。例如，UserDaoTest或StringUtilTest。

3.3 方法名

  • 方法名都以小驼峰式命名法风格编写。

  • 方法名通常是动词或动词短语。如addParameter()、login()、getUserInfo()。

  • 测试方法名要求以test开头，如testLogin()、testGetUserInfo();

3.4 常量名

常量名应全部字母大写，用下划线分隔单词，如USER_NAME，NETWORK_STATE。

// 常量
static final int NUMBER = 5;
static final ImmutableList<String> NAMES = ImmutableList.of("Ed", "Ann");
static final Joiner COMMA_JOINER = Joiner.on(',');  // because Joiner is immutable
static final SomeMutableType[] EMPTY_ARRAY = {};
enum SomeEnum { ENUM_CONSTANT }

// 非常量
static String nonFinal = "non-final"; 
final String nonStatic = "non-static";
static final Set<String> mutableCollection = new HashSet<String>();
static final ImmutableSet<SomeMutableType> mutableElements = ImmutableSet.of(mutable);
static final Logger logger = Logger.getLogger(MyClass.getName());
static final String[] nonEmptyArray = {"these", "can", "change"};

这些名字通常是名词或名词短语。

3.5 非常量字段名

  • 非常量字段名以小驼峰式命名法风格编写。

  • 这些名字通常是名词或名词短语。

3.6 参数名

  • 非常量字段名以小驼峰式命名法风格编写。

  • 参数应该避免用单个字符命名。

  • 暴露给前端的参数以下划线法命名法命名，内部使用参数以小驼峰式命名法命名。

3.7 局部变量名

  • 局部变量名以小驼峰式命名法风格编写。

  • 避免用单字符进行命名，除了临时变量和循环变量。

  • 即使局部变量是final和不可改变的，也不应该把它示为常量，自然也不能用常量的规则去命名它。

3.8 类变量名

类变量名以小驼峰式命名法风格编写，如count，userNameTextView。

public class UserService {
   // 类变量名
   private UserDao userDao = new UserDaoImpl();
   private int count;

   public boolean login(String userName, String password) {
        boolean result = true;
        if(userDao.getUser(userName, password) != null) {
            return true;
        }
        return false;
   }
}

4 格式

4.1 使用大括号(即使是可选的)

大括号与if, else, for, do, while语句一起使用，即使只有一条语句(或是空)，也应该把大括号写上。

4.2 非空块：K & R 风格

对于非空块和块状结构，大括号遵循Kernighan和Ritchie风格 (Egyptian brackets)：

  • 左大括号前不换行
  • 左大括号后换行
  • 右大括号前换行
  • 如果右大括号是一个语句、函数体或类的终止，则右大括号后换行; 否则不换行。例如，如果右大括号后面是else或逗号，则不换行。

示例：

return new MyClass() {
  @Override public void method() {
if (condition()) {
   try {
 something();
   } catch (ProblemException e) {
 recover();
   }
  }
}
};

4.3 空块：可以用简洁版本

一个空的块状结构里什么也不包含，大括号可以简洁地写成{}，不需要换行。

示例：

void doNothing() {}

例外：如果它是一个多块语句的一部分(if/else 或 try/catch/finally) ，即使大括号内没内容，右大括号也要换行。

4.4 列限制：120个字符

要求每行代码最大字符数为120个字符，除了下述例外，任何一行如果超过这个字符数限制，必须自动换行。

例外：

  • 不可能满足列限制的行(例如，Javadoc中的一个长URL，或是一个长的JSNI方法参考)。
  • package和import语句(见3.2节和3.3节)。
  • 若使用Eclipse，可在工具栏中依次点击Windows -> Preferences -> Code Style -> Formatter ->Edit -> Line Wrapping -> 在Maximum line width中填写列字符限制长度，选择Apply或OK即可。当使用“代码自动格式化（Ctrl + Shift + F）”后Eclipse会自动根据行字符限制对代码进行换行处理。

4.5 空白

4.5.1 垂直空白

以下情况需要使用一个空行：

  1. 类内连续的成员之间：字段，构造函数，方法，嵌套类，静态初始化块，实例初始化块。
  2. 例外：两个连续字段之间的空行是可选的，用于字段的空行主要用来对字段进行逻辑分组。
  3. 在函数体内，语句的逻辑分组间使用空行。
  4. 类内的第一个成员前或最后一个成员后的空行是可选的(既不鼓励也不反对这样做，视个人喜好而定)。
  5. 要满足本文档中其他节的空行要求(比如import语句)
4.5.2 水平空白

除了语言需求和其它规则，并且除了文字，注释和Javadoc用到单个空格，单个ASCII空格也出现在以下几个地方：

  1. 分隔任何保留字与紧随其后的左括号(()(如if, for catch等)。
  2. 分隔任何保留字与其前面的右大括号(})(如else, catch)。
  3. 在任何左大括号前({)，两个例外：
    • @SomeAnnotation({a, b})(不使用空格)。
    • String[][] x = {{"1","2","3"},{"4","5","6"}};(大括号间没有空格)。
  4. 在任何二元或三元运算符的两侧。这也适用于以下“类运算符”符号：
    • 类型界限中的&(<T extends Foo & Bar>)。
    • catch块中的管道符号(catch (FooException | BarException e)。
    • foreach语句中的分号。
  5. 在, : ;及右括号())后。
  6. 如果在一条语句后做注释，则双斜杠(//)两边都要空格。这里可以允许多个空格，但没有必要。
  7. 类型和变量之间：List list。
  8. 数组初始化中，大括号内的空格是可选的，即new int[] {5, 6}和new int[] { 5, 6 }都是可以的。(最好加上空格)

5 注释

写注释的目的是为了增加代码的可读性，便于后期维护，并不是所有的代码块都需要添加注释。

若使用Eclipse，可对注释模板进行配置。设置注释模板的入口： Window -> Preference -> Java -> Code Style -> Code Template，然后展开Comments节点，就是所有需要设置注释模板的元素了（此步骤以下简称“打开设置注释模板界面”），具体注释要求和模板内容，下面会进行阐述。

5.1 类注释风格

要求每个类必须写明类的功能描述、编写者、编写时间、版本，若该类被修改请及时更新这些信息。如下：

 /**
  * 车辆配置Activity
  * @date 2015-6-28 上午10:41:41 
  * @version V2.5
  */
public class CarConfigurationActivity extends BaseActivity {
   // ...
}

Eclipse注释模板设置：打开设置注释模板界面选择Type进行编辑，设置内容如下：

/** 
 * @date ${date} ${time} 
 * @version V1.0
 */

5.2 成员属性注释风格

直接在类成员属性上方或结束处使用“//”进行注释即可，需注意“//”前后空格。

示例：

public class Response<T> implements Serializable {

private int status; // 状态码
private String message; // 请求结果信息
private T response; // 请求返回的数据

// ...
}

5.3 一般方法注释风格

需写明方法的功能描述、参数含义、返回值含义、异常信息（如无异常抛出则不需写）。

示例：

/*
 * 添加需提交的参数对
 * @param params 参数对
 * @return 当前对象
 * @throws Exception
 */
public RequestClient addParameter(String... params) throws Exception {

//...
}

Eclipse注释模板设置：打开设置注释模板界面选择Methods进行编辑，设置内容如下：

/**
 * ${tags}
 */

5.4 构造方法注释风格

只需指定方法的功能描述为Constructor即可，其它参考一般方法注释风格。

Eclipse注释模板设置：打开设置注释模板界面选择Methods进行编辑，设置内容如下：

/**
 * Constructor
 * ${tags}
 */

5.5 普通块代码注释

直接在类成员属性上方或结束处使用“//”进行注释即可。

6 其它

6.1 switch语句中default的情况要写出来

每个switch语句都必须包含一个default语句组，即使它什么代码也不包含。

6.2 捕获的异常：不能忽视（打印+添加注释）

除了下面这种例子，对捕获的异常不做响应是极少正确的。(典型的响应方式是打印日志，或者如果它被认为是不可能的，则把它当作一个AssertionError重新抛出。)

如果它确实是不需要在catch块中做任何响应，需要做注释加以说明(如下面的例子)。

try {
  int i = Integer.parseInt(response);
  return handleNumericResponse(i);
} catch (NumberFormatException ok) {
  // it's not numeric; that's fine, just continue
}
return handleTextResponse(response);

例外：在测试中，如果一个捕获的异常被命名为expected，则它可以被不加注释地忽略。下面是一种非常常见的情形，用以确保所测试的方法会抛出一个期望中的异常， 因此在这里就没有必要加注释。

try {
  emptyStack.pop();
  fail();
} catch (NoSuchElementException expected) {
}

6.3 @Override注解

只要是合法的，就把@Override注解给用上。

6.4 静态成员：使用类进行调用

使用类名调用静态的类成员，而不是具体某个对象或表达式。

示例：

Foo aFoo = ...;
Foo.aStaticMethod(); // good
aFoo.aStaticMethod(); // bad
somethingThatYieldsAFoo().aStaticMethod(); // very bad

6.6 返回数据状态码

后台返回状态码不能随便写，必须遵循以下规则

  • 公共状态如成功("success")必须是1000，失败("failure")必须是1100，其他新增加的公共状态码须在 在1000到1999之间。

  • 处理异常用状态须在2000到2999之间。

  • 用户相关状态须在3000到3999之间。

  • 项目业务相关状态须在4000到4999之间

具体示例如下：

//In common use
SUCCESS(1000, "success"),
FAILURE(1100, "failure"),

//Handle with ExceptionHandler

INSERT_DUPLICATE(2011, "insert repeat"),
AUTHENTICATION_ERROR(2012,"authentication_error"),  
INVALID_PARAMETER(2020,"invalid_parameter"),
INVALID_TOKEN(2100,"Invalid token"),
INTERNAL_SERVER_ERROR(2222,"internal_server_error"),

//User Part
USER_NOT_EXIST(3001, "user not exist"),

CODE_TIMEOUT(3102,"verificationCode is overdue"),

6.7 API文档编写

  • 接口返回数据尽量不要出现null。

  • 每个接口会出现的状态必须注明清楚状态码、状态码说明。

  • 参数是否必须要明确表明，特殊情况要特别说明。

  • 每次更新文档必须更新文档更新记录，什么时间谁修改了什么，什么时间谁添加了什么等。

  • 特定的参数须注明，如machine_status——机器状态(on/off)、再如detail_json是json格式字符串须有如下提示

6.8 代码中必须要有必要的LOG

  • log要显示类名、方法名、简要的说明和一些必要的数据。

  • log要醒目，在log文件中要很清楚的看出。

  • 在方法的开始、成功结束、失败结束必须有log记录，一些逻辑比较复杂的需看具体情况增加log记录。
